문제 1 · Ch1: HIPM & Big Three (70점)

상황: 최근 출시된 차량용 인포테인먼트 터치스크린 시스템 조작 중 전방 주시 태만으로 인한 추돌 사고가 급증하고 있다. 제조사는 터치 반응 속도를 높였음에도 사고율 감소 효과를 얻지 못했다. "내비게이션 조작 중 추돌"과 "음악 조작 중 추돌"은 조작 행위는 다르지만, 분석에서는 둘 다 시선 분산이라는 상위 범주로 묶을 수 있다.

문항 1-1. 인간 정보 처리 모형(Human Information Processing Model; Wickens et al., 2022)의 4단계를 적용하여, 현재 시스템의 병목(Bottleneck)이 발생하는 단계를 특정하고 그 인지적 원인을 분석하시오. (30점)
문항 1-2. 해당 시스템의 UI를 인간 수행의 3대 지표 중 '정확성(Accuracy)'과 '주의 요구량(Attentional Demand)' 측면에서 정량적으로 평가하기 위한 실험 설계(독립변인, 종속변인, 측정 방법)를 구체적으로 제안하시오. (40점)

문항 1-1 전략 — 병목 단계 특정

핵심 열쇠: "터치 반응속도를 높였는데 사고가 안 줄었다" → 제조사는 Response Execution 단계만 건드렸다. 병목은 훨씬 앞에 있다.

HIPM 4단계 매핑표

단계기능이 시나리오에서
① 감각처리 + STSS시각·청각 신호 수용 (~1초 유지)전방 도로 + 화면 → 두 시각 신호 동시 입력
② 지각 (Perception)감각 신호에 의미 부여 (LTM 기반)병목 ①: 전방 도로 지각과 화면 아이콘 지각이 동일 시각 자원 경쟁
③ 작업기억/인지정보 일시 보관 + 의사결정 준비병목 ②: 내비 경로 + 운전 상황 동시 유지 → WM 용량 초과
④ 반응선택 → 반응실행행동 선택 + 근육 제어제조사가 개선한 부분 — 병목이 앞 단계에 있으면 무의미

병목 논증 구조

주장: 병목은 ② 지각(Perception) 단계의 Attention Filter 실패
      + ③ 작업기억(WM) 자원 고갈

논거:
Attention = Filter(선택) + Fuel(자원) — Wickens & McCarley (2008)
운전 → 시각 공간적 주의를 지속 요구 → Filter가 전방 고정
터치스크린 → 동일한 시각 공간 채널 요구
⟹ 두 과제가 같은 Fuel을 경쟁 → 자원 고갈 → 지각 오류 발생
Response Execution 개선은 이 자원 경쟁을 해결하지 못함
형모님이 챗봇 UI를 설계할 때 "경찰관이 피해자 면담을 하면서 챗봇 화면도 봐야 한다"는 상황과 동일한 병목이다. 면담(청각)과 챗봇(시각)은 모달리티가 달라 덜 심각하지만, 운전+터치스크린은 시각 vs 시각이라 치명적이다.

문항 1-2 전략 — 실험 설계

Accuracy와 Attentional Demand는 측정법이 다르다. 각각 IV·DV·측정법을 명확히 분리해서 서술해야 한다.

설계 요소정확성(Accuracy)주의요구량(Attentional Demand)
독립변인(IV)UI 복잡도 (단순 3버튼 / 계층 메뉴), 과제 유형 (내비 / 음악 / 에어컨)동일
종속변인(DV)오조작 횟수, 정확 조작률, 완료 시간차선 이탈 빈도, 앞차 거리 변동, 이중과제 RT 증가량
측정 방법드라이빙 시뮬레이터 + 조작 로그 자동 기록Eye-tracking (시선 이탈 지속시간), NASA-TLX, 보조과제 청각 RT
기대 결과복잡 UI → 오조작률 증가복잡 UI → 차선이탈↑, Attentional Demand 증가
★ 답안 구조 제안 (1-2) 실험 목적 1문장 → 참가자 기준 → Accuracy 설계(IV/DV/측정법) → Attentional Demand 설계(IV/DV/측정법) → 두 지표 간 trade-off 언급으로 마무리

문제 2 · Ch2: SDT, d'/β, Likelihood Alarms (70점)

상황: A 기업은 AI 기반 보안 모니터링 시스템을 도입하였다. 미탐(Miss) 방지를 위해 매우 낮은 임계값을 유지하도록 설계되어 빈번한 오경보(False Alarm)를 발생시키고 있다. 보안 관리자는 초기에 엄격히 대응했으나, 오경보 누적으로 경보를 불신하게 되었으며, 긴급한 실제 위협 경보조차 "단순 오류"로 무시하는 양치기 소년 효과(Cry wolf syndrome; Wickens et al., 2022)를 보이고 있다.

문항 2-1. 관리자의 행동 변화를 신호 탐지 이론(SDT)의 민감도(d')와 판단 기준(β) 개념을 설명하고 이를 활용하여 수리적·인지적 기제로 설명하시오. 특히 오경보 남발이 d'를 변화시킨 것인지, 아니면 β의 이동을 유발한 것인지 핵심 기제를 명확히 논하시오. (35점)
문항 2-2. Sorkin(1989)의 연구에 기반하여 이분법적(Binary) AI 알람 방식을 대체할 수 있는 우도 알람(Likelihood Alarms)의 개념을 정의하시오. 또한 AI 분석 위험 확률에 따라 관리자가 직관적으로 대응 우선순위를 판단할 수 있도록, 시각적·청각적 요소를 결합한 구체적인 'AI 보안 관제 인터페이스' 설계안을 제시하시오. (35점)

문항 2-1 전략 — 핵심 논쟁: d' vs β

결론 먼저: 오경보 남발이 변화시킨 것은 β(판단 기준)의 이동이다. d'는 변하지 않는다.

개념 정의

개념정의이 시나리오에서
d' (민감도)신호 분포와 잡음 분포의 평균 거리 (표준편차 단위)AI 경보 시스템의 신호 품질 — 시스템이 설계된 이상 관리자 경험으로 바뀌지 않음
β (판단 기준)X_C 지점에서 신호분포 높이 / 잡음분포 높이의 비율관리자가 "경보에 반응할 것인지" 결정하는 내부 임계값 → 학습으로 이동 가능

d'는 변하지 않는다 — 논거

d'는 AI 시스템이 진짜 위협과 정상 업무 활동을 통계적으로 구분하는 물리적 품질이다. 관리자의 경험 축적과 무관하게, 시스템의 임계값(threshold)이 낮게 설정된 이상 신호/잡음 분포의 분리도 자체는 변하지 않는다 (Wickens et al., 2022).

β가 이동하는 수리적·인지적 기제

초기 (오경보 드묾):
  관리자 β 낮음 (liberal) → 경보 울리면 즉각 대응

오경보 누적 후:
  P(실제위협 | 경보) 가 매우 낮다고 학습됨
  → 주관적 P(S) ↓
  → β_opt = P(N)/P(S) × [V(CR)+C(FA)] / [V(H)+C(M)]
  → P(N)/P(S) 급증 → β_opt 상승
  → 관리자 β 우측 이동 (conservative)
  → 진짜 위협 경보도 기준 아래 → Miss 발생

ROC 곡선 해석:
  d' 동일한 등감도선(isosensitivity curve) 위에서
  오른쪽으로 이동 → P(H)↓ P(FA)↓ 동시 발생
  (각성이론의 beta 증가 패턴과 동일한 표면 현상)
형모님의 피해자전담경찰봇 Tier 모델 — "Tier 4 위반 금지"를 설정한 것이 바로 β 관리 전략이다. 챗봇이 위기를 너무 자주 탐지하면 경찰관이 무시하기 시작한다. 동일한 Cry wolf 메커니즘.

문항 2-2 전략 — Likelihood Alarms 설계

Binary 방식Likelihood Alarms (Sorkin, 1989)
정보 형태경보 / 비경보 (0 또는 1)위험 확률 연속값 (예: 20%, 60%, 95%)
관리자 행동β 하나로만 반응 → 최적 대응 불가확률 수준별로 다른 β 적용 → 최적 행동 가능
Cry wolf 해결낮은 임계값 → 모든 경보 동일 취급 → 무시낮은 확률 경보는 무시, 높은 확률만 대응

AI 보안 관제 인터페이스 설계안

시각 요소:
  - 색상 그라디언트: 녹색(~20%) → 황색(20~50%) → 주황(50~80%) → 적색(80%+)
  - 경보 카드 크기: 위협 확률 비례하여 증가
  - 확률 수치 명시: "위협 확률 87%" 직접 표시
  - 히스토리 표시: 동일 IP 반복 이상 행동 이력 병기

청각 요소:
  - 20% 미만 → 무음
  - 50% 이상 → 낮은 단음 알림
  - 80% 이상 → 높은 주파수 반복 경고음 + 음성("즉각 확인 요망")

레이아웃:
  - 높은 확률 경보 → 화면 중앙/상단 (Salience↑, Effort↓)
  - 낮은 확률 경보 → 사이드바 격리

문제 3 · Ch3: SEEV, PCP (70점)

상황: 연세대학교의 learnUs 플랫폼은 강의 자료 공유, 과제 제출, 실시간 알림 등 방대한 기능을 제공하고 있다. 그러나 최근 대학원생과 교수들이 중요한 학사 경고나 과제 마감 알림, 긴급 공지사항을 확인하지 못해 행정적 차질을 빚는 사례가 빈번하다. 사용자들은 "화면에 정보가 너무 많아 무엇이 중요한지 모르겠다"거나 "알림이 떴는지조차 몰랐다"는 피드백을 제시하고 있다.

문항 3-1. 공학심리학 전공자로서 learnUs의 인터페이스를 LMS로서 UX와 기능을 분석 평가하시오. 또한 사용자가 "중요 알람을 놓치는 현상"을 SEEV(Salience, Effort, Expectancy, Value) 모델의 4가지 요인을 기준으로 분석하고, 이를 상향식(Bottom-up)과 하향식(Top-down) 기제로 분류하여 논리적으로 설명하시오. (35점)
문항 3-2. Wickens와 Carswell(1995)이 제안한 근접 호환성 원칙(PCP)에 근거하여, 사용자가 학습 콘텐츠에 집중하고 있는 전방 주시(Focused attention) 상태를 방해하지 않으면서도 시스템 경고를 즉각 인지할 수 있도록 돕는 '시각적 유도 및 디스플레이 설계 방안'을 구체적으로 제시하시오. UX, UI 디자인을 고려하여 기능을 어떻게 업그레이드하는 게 대학원생들에게 활용을 잘하게 만드는지 의견을 제시하시오. (35점)

문항 3-1 전략 — SEEV 분석

SEEV 공식 먼저 제시:
SEEV = wS·Salience + wE·(−Effort) + wEx·Expectancy + wV·Value

Bottom-up / Top-down 분류 — 채점 핵심

요인방향learnUs 문제
S (Salience)Bottom-up알림 배지 작음, 깜빡임 없음, 강의 콘텐츠와 색상 대비 낮음 → 시각적 돌출 실패
E (Effort)Bottom-up알림 아이콘 위치 = 화면 우상단 → 시선 중심(콘텐츠)에서 각도 이탈 큼 → 이동 비용↑
Ex (Expectancy)Top-down평소 알림이 드물고 대부분 덜 중요 → 중요 알림이 올 것이라는 기대 낮음
V (Value)Top-down이전 알림 클릭 시 "별 내용 없었다"는 반복 경험 → 알림에 할당하는 주관적 가치 낮음
learnUs UX 서두 추천: "learnUs는 강의자료 열람, 과제 제출, 공지 알림을 단일 플랫폼에 통합한 LMS로서 기능적 완결성은 높으나, 정보 밀도 과다(information overload)와 알림 계층 미분화로 인해 다음과 같은 인지공학적 문제를 보인다."
형모님은 learnUs 대학원생 사용자다. "과제 마감 하루 전, 공지사항 5개 중 어느 것이 마감 알림인지 바로 보이지 않았다" — 이것이 S(Salience) 실패 + V(Value) 저하의 실제 사례다. 직접 경험 쓰면 된다.

문항 3-2 전략 — PCP 설계

PCP 원리 먼저 정의: 동시에 처리해야 하는 정보는 디스플레이 상에서 근접하게 배치해야 한다. 전방 주시 과제에서는, 경고 정보를 초점 영역에 통합하거나 주변시(peripheral vision)로 인지 가능하게 설계해야 한다 (Wickens & Carswell, 1995).

구체 설계 방안

1. 콘텐츠 근접 경고 오버레이 (PCP 핵심)
   강의 영상/PDF 화면 모서리에 긴박도 바 삽입
   → 초점 이동 없이 주변시로 경고 수준 인지
   → "마감 2시간 전" → 상단 가장자리 오렌지 바 등장

2. Ambient 주변시 알림 (Focal 방해 없이)
   화면 전체 테두리가 서서히 색 변화 (무채색 → 주황 → 적색)
   → 초점 콘텐츠 유지 + 주변시가 변화 감지
   → Salience↑, Effort 최소화

3. 중요도 계층 분리 (Value 신뢰 회복)
   긴급(학사경고, 24h 이내 마감) vs 일반(강의자료 업로드) 완전 분리
   → 긴급 = 전체화면 팝업 / 일반 = 사이드바
   → 알림 클릭 = 항상 중요한 내용 → Value 기대 상승

대학원생 활용도 향상 UX 의견

  • 맞춤 알림 설정: 수강 과목별 마감일 자동 캘린더 연동
  • 행동 기반 우선순위: 사용자가 자주 놓치는 카테고리 학습 → 자동 Salience 부스트
  • 모바일 연동 강화: 푸시 알림 → desktop보다 눈에 잘 띔, Effort 최소화
  • 원클릭 확인 처리: 현재 3단계 클릭 → 1단계로 축소 (Effort 비용 절감)
  • 알림 신호음 개인화: 긴급 공지만 소리 알림, 나머지 무음 → Expectancy 교정
챗봇 UI에서 경찰관에게 "긴급 면담 필요" 알림을 전달하는 문제와 동일하다. 전문가 사용자(경찰관, 대학원생)는 핵심 작업 중 방해 싫어하지만 중요 신호는 놓치면 안 된다 — 이 긴장이 PCP 설계의 핵심이다.
